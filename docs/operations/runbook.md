# PFA Cage Rentals — runbook

Operational procedures for keeping the production site at
[pfaengine.com](https://pfaengine.com) healthy. Each section
should be runnable end-to-end without asking the original author a
question.

If you're new to this codebase, start with [README.md](../../README.md)
and [BRAINSTORM.md](../../BRAINSTORM.md) for context, then come back here
when something goes sideways.

---

## Site down — diagnostic order

Work through these in order. Stop as soon as one is the obvious cause.

1. **Vercel status** — [vercel-status.com](https://www.vercel-status.com).
   If Vercel is degraded, there's no fix from our side. Subscribe to
   incident updates and post the link in any user-facing comms.
2. **Vercel deployment** — open the project in the Vercel dashboard.
   Latest deployment should be green. If the most recent push failed,
   the previous deployment is still serving — the site is "stale" but
   not "down."
3. **Neon database** — [neonstatus.com](https://neonstatus.com), then
   the Neon console for our project. A Neon outage shows up as 500s on
   every authenticated route (Auth.js can't load the session row).
4. **DNS at GoDaddy** — `dig pfaengine.com +short` should resolve
   to Vercel's anycast IPs. If DNS is broken, log in to GoDaddy (NOT
   Vercel — the domain lives at GoDaddy, see project memory) and check
   the CNAME for `@` and `www`.
5. **CSP block in Sentry** — open Sentry → Issues, filter by
   `CSP violation`. A misconfigured Content-Security-Policy can block
   the OAuth redirect or the Resend magic-link form.
6. **Rate limit triggered** — if sign-in specifically is broken, check
   `src/lib/ratelimit.ts` thresholds and the Upstash dashboard for the
   `magic-link:email` and `magic-link:ip` buckets. A burst of sign-in
   attempts can lock out everyone on the same NAT'd IP for an hour.

If none of those apply, walk the request path manually: browser →
Vercel Edge → Next.js server function → Auth.js or business action →
Drizzle → Neon. Read `src/db/schema.ts` for the data model and
`docs/process/production-checklist.md` for stage history. The dedicated
`docs/reference/architecture.md` (K3 in the checklist) is also up.

## Restore from PITR (Neon point-in-time recovery)

Neon keeps a 7-day branch history by default. To roll back:

1. Neon console → project → **Branches** → **Create branch**.
2. Source = `main`, "Restore branch to a specific time" = the timestamp
   *just before* the bad change (UTC).
3. Name the new branch something like `recovery-2026-05-25T01-00Z`.
4. Copy the new branch's connection string.
5. Update `DATABASE_URL` in Vercel → Project Settings → Environment
   Variables (Production). Redeploy (see "Trigger redeploy" below).
6. Once verified, promote the recovery branch to be the new `main` via
   Neon's "Set as default" action, or keep both and migrate forward.

The integration test branch (pointed at by `INTEGRATION_DATABASE_URL`)
is separate and doesn't need recovery — it's regenerated by `npm run
db:migrate && npm run db:seed`.

## Restore from nightly backup (Cloudflare R2)

The PITR window above only covers the last 24 hours on Neon's free
tier. Beyond that, the canonical recovery path is the nightly
`pg_dump` written to Cloudflare R2 by `.github/workflows/backup.yml`.
The bucket has a 30-day lifecycle rule so each file is around for
roughly a month.

To restore:

1. Pick the dump you need. From R2's dashboard or via rclone:
   ```bash
   rclone ls r2:pfa-cage-rentals-backups/nightly/ | sort
   ```
2. Download it locally:
   ```bash
   rclone copy r2:pfa-cage-rentals-backups/nightly/pfa-cage-rentals-2026-05-25T06-00-00Z.sql.gz .
   gunzip pfa-cage-rentals-2026-05-25T06-00-00Z.sql.gz
   ```
3. Create a fresh Neon branch (Neon console → Branches → New) so the
   restore goes to a side-channel before you cut over. Copy its
   connection string.
4. Replay the dump:
   ```bash
   psql "<recovery branch connection string>" < pfa-cage-rentals-2026-05-25T06-00-00Z.sql
   ```
5. Smoke the recovery branch by setting it as `DATABASE_URL` locally
   and running `npm run dev`. When confident, swap Vercel's
   `DATABASE_URL` to the recovery branch and redeploy.

The dumps include the schema + data + extension definitions
(`btree_gist`). You don't need to run migrations against the restore
target first.

### Provisioning the backup pipeline

If the workflow has never been wired up (or you're moving R2 accounts):

1. Cloudflare dashboard → R2 → Create bucket `pfa-cage-rentals-backups`. Set the bucket's **Object Lifecycle Rule** to delete objects older than 30 days.
2. R2 → Manage R2 API Tokens → create a token with read+write to that bucket only. Copy the access key id + secret + account id.
3. GitHub repo → Settings → Secrets and variables → Actions → New repository secret. Add: `DATABASE_URL_PROD` (Neon prod pooler URL), `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET` (just `pfa-cage-rentals-backups`).
4. Manually run the workflow once: GitHub → Actions → "Nightly backup" → Run workflow. Verify the dump appears under `nightly/` in R2.

## Rotate a leaked secret

Always rotate the **provider side first**, then update Vercel and
redeploy. If you change Vercel first, the old secret is still issuing
tokens until you invalidate it upstream.

### `AUTH_SECRET`
1. Generate a new value: `openssl rand -base64 32`.
2. Set it in Vercel (Production + Preview + Development scopes).
3. Redeploy. Every existing session cookie becomes invalid — users
   re-sign-in. Communicate this in #pfa-ops if it's during business
   hours.

### `AUTH_GOOGLE_ID` / `AUTH_GOOGLE_SECRET`
1. Google Cloud Console → APIs & Services → Credentials → our OAuth
   2.0 Client → "Reset secret." This invalidates the old secret
   immediately.
2. Copy the new client ID + secret. Update Vercel.
3. Redeploy. No user impact unless they sign in mid-rotation (they get
   a one-time error and a retry works).

### `AUTH_RESEND_KEY`
1. resend.com → API Keys → revoke the leaked key.
2. Create a new key, scoped to "Sending access" only.
3. Update Vercel → redeploy.
4. Email sender (`noreply@pfaengine.com`) and the verified domain
   stay configured on Resend's side — only the API key rotates.

### `UPSTASH_REDIS_REST_URL` / `UPSTASH_REDIS_REST_TOKEN`
1. console.upstash.com → our DB → "REST API" tab → "Reset Token."
2. Update Vercel → redeploy.
3. Rate-limit counters reset along with the token — a malicious actor
   who hit the limit before rotation no longer is.

### `DATABASE_URL`
1. Neon console → project → **Roles** → rotate the password on the
   service role (or create a fresh role + grant + delete the old).
2. Update Vercel's `DATABASE_URL` → redeploy.
3. Don't forget integration: update `INTEGRATION_DATABASE_URL` in
   GitHub Actions secrets too, if the leak involved that role.

### Sentry (`NEXT_PUBLIC_SENTRY_DSN`, `SENTRY_AUTH_TOKEN`)
1. Sentry → Settings → Projects → pfa-cage-rentals → **Client Keys
   (DSN)** → "Disable" the old key, create a new one.
2. For the build-time auth token: Settings → Auth Tokens → revoke +
   create new with scopes `project:read`, `project:releases`,
   `org:read`.
3. Update Vercel → redeploy. Source-map upload on the next deploy uses
   the new token; client error reporting picks up the new DSN.

## Add a new coach manually

Coaches normally appear when they first sign in (Auth.js creates the
user row). To add one before they've signed in (e.g. for testing,
or so they show up in the picker before their first login):

```sql
-- replace email + name as appropriate
INSERT INTO users (id, email, name, role)
VALUES (gen_random_uuid(), 'newcoach@example.com', 'New Coach', 'coach');
```

Run this against the production `DATABASE_URL` from the Neon SQL
editor, or locally:

```bash
DATABASE_URL="<prod url>" npx tsx -e "
import { config } from 'dotenv';
config({ path: '.env.local' });
import { neon } from '@neondatabase/serverless';
const sql = neon(process.env.DATABASE_URL);
await sql\`INSERT INTO users (email, name, role) VALUES ('newcoach@example.com', 'New Coach', 'coach')\`;
"
```

We don't audit-log manual inserts — they're admin-only and rare. If
that becomes a problem, write the audit row by hand to `audit_log`
with `entity_type='user'`, `action='create'`, and `actor_user_id` =
your admin id.

## Trigger a redeploy

The cheapest way:

```bash
git commit --allow-empty -m "trigger redeploy"
git push origin main
```

Or via Vercel UI: Project → Deployments → "..." on the latest →
**Redeploy**. The UI option re-runs the build *with the same source*,
which is correct for env-only changes (e.g. after rotating a secret).
The empty-commit path re-fetches from `main` and runs the full
pipeline, which is what you want if anything in the repo has changed.

## Billing dispute — pull the audit trail

A coach reports they were billed for a session they didn't have, or
the rate looks wrong. The audit log has every change. Walk through:

1. **Find the session row** — `/admin/sessions` or by querying:
   ```sql
   SELECT * FROM sessions_billing
   WHERE coach_id = (SELECT id FROM users WHERE email='coach@example.com')
     AND start_at >= '2026-05-01' AND start_at < '2026-06-01'
   ORDER BY start_at;
   ```
2. **Pull the audit history** for that session id:
   ```sql
   SELECT a.ts, u.email AS actor, a.action, a.diff
   FROM audit_log a JOIN users u ON u.id = a.actor_user_id
   WHERE a.entity_type = 'session' AND a.entity_id = '<session uuid>'
   ORDER BY a.ts;
   ```
   `diff` shows the full before/after snapshot for creates and deletes,
   or the changed keys for updates.
3. **Rate at the time of billing** — `sessions_billing` snapshots the
   rate at the moment of creation. If the coach's override has
   changed since, the OLD rate still applies to the old session. The
   per-coach override history is in `audit_log` with `entity_type =
   'rate_override'` and `entity_id = '<coachId>:<resourceType>'`.

The audit log is append-only — there's nothing to repair, only to
read.

## Onboard a new admin

The admin allowlist is **hardcoded** in `src/lib/admin-emails.ts`, not
an environment variable. To promote a new admin:

1. Add their email (lowercase) to `ADMIN_EMAILS` in
   `src/lib/admin-emails.ts`.
2. Commit + push to `main` (Vercel auto-deploys).
3. Ask the new admin to sign in. The `createUser` event in
   `src/auth.ts` checks `isAdminEmail()` and updates `role` to `admin`
   on first sign-in.
4. If they've already signed in as a coach, the role won't auto-
   upgrade on subsequent sign-ins — flip them by hand:
   ```sql
   UPDATE users SET role = 'admin' WHERE email = 'newadmin@example.com';
   ```

Removing an admin is the same flow in reverse: drop them from
`ADMIN_EMAILS`, push, and run `UPDATE users SET role = 'coach' WHERE
email = '...'`. Their session cookie stays valid until they sign out
or it expires — if you need immediate revocation, also
`DELETE FROM sessions WHERE user_id = '<their id>'`.

## Delete a coach (account deletion request)

Coaches can request deletion per the Privacy Policy. The admin UI on
`/admin/coaches/[id]` has a **Danger zone** card with a typed-
confirmation modal. That's the canonical path — it soft-deletes the
user row (anonymizes name + email, sets `deletedAt`), revokes their
session + OAuth account, and writes a `delete` audit entry.

You should not need to do this by hand. If for some reason you do:

```sql
-- 1) capture the original identity into the audit log (otherwise
--    the anonymized row replaces the trail)
INSERT INTO audit_log (actor_user_id, entity_type, entity_id, action, diff)
SELECT '<your admin uuid>', 'user', id, 'delete',
       jsonb_build_object('before', row_to_json(u.*))
FROM users u WHERE u.id = '<target uuid>';

-- 2) anonymize the user row
UPDATE users
SET deleted_at = NOW(),
    name = 'Former coach',
    email = 'deleted-' || id || '@pfacagerentals.invalid',
    image = NULL
WHERE id = '<target uuid>';

-- 3) revoke active sessions + linked accounts + magic-link tokens
DELETE FROM sessions WHERE user_id = '<target uuid>';
DELETE FROM accounts WHERE user_id = '<target uuid>';
-- (verification_tokens are keyed by the original email — capture it
--  before step 2 if you want to clean those up too)
```

But really, just use the UI.
